📌 Introducción

Sistema de monitoreo dual-WAN para MikroTik RouterOS que detecta cortes de fibra/ISP en tiempo real, actualiza registros DDNS en cdmon.org y envía alertas vía Telegram/WhatsApp. Existen 3 versiones según la topología de la WAN2 (la WAN1 siempre es IP estática).

✨ Características comunes a las 3 versiones

  • ✅ Detección en 2 etapas: enlace local + internet real
  • ✅ Failover automático con routing recursivo (WAN1)
  • ✅ Alertas en tiempo real: Telegram + WhatsApp
  • ✅ DDNS solo se actualiza si la IP cambió (consulta previa a cdmon)
  • ✅ Log limpio en producción (debugEnabled false)
  • ✅ Soporta hardware MIPS mono-core (RB2011UiAS-2HnD)

⚙️ Setup de Routing Recursivo (común a las 3)

¿Por qué es necesario?

En failover por distancia, si WAN1 pierde fibra/ISP pero el modem sigue vivo, el check-gateway=ping al modem responde OK y la ruta NO failovea. El router sigue usando una WAN caída.

Solución: Una ruta sonda recursiva que pinguea un host remoto a través de la fibra. Si la fibra cae, 1.1.1.1 no responde → la ruta se inactiva → failover real.

Comandos (ajustar IPs por equipo)

# Sonda: 1.1.1.1 sale por el modem WAN1
/ip route add dst-address=1.1.1.1/32 gateway=192.168.10.1 scope=10 comment="WAN1 probe recursivo"

# Default recursiva (CRÍTICO: target-scope=11 en ROS 7.20.7)
/ip route add dst-address=0.0.0.0/0 gateway=1.1.1.1 target-scope=11 distance=1 check-gateway=ping comment="WAN1 default recursivo"
⚠️ target-scope=11 es CRÍTICO en RouterOS 7.20.7. Otras versiones pueden requerir 10 o 0. Verificá con /ip route print where gateway="1.1.1.1" que muestre estado A (Active).

Routing extra para WAN2 (solo V2 y V3)

Para que el fetch/ping de WAN2 salga por su interface aunque esté en standby:

# V2 (IP fija WAN2)
/routing table add name=to-wan2 fib
/ip route add dst-address=0.0.0.0/0 gateway=192.168.20.1 routing-table=to-wan2
/routing rule add src-address=192.168.20.254 action=lookup table=to-wan2

# V3 (Starlink DHCP) - tabla salida_WAN2
/routing table add name=salida_WAN2 fib
/ip route add dst-address=0.0.0.0/0 gateway=GATEWAY_STARLINK routing-table=salida_WAN2 check-gateway=ping
# Nota: la IP de Starlink es dinámica; la rule debe ajustarse al rango asignado

🔧 Configuración del Script

Todas las versiones tienen una ZONA DE CONFIGURACIÓN al inicio. Variables principales:

# Debug: true = ves todo | false = solo alertas reales (PRODUCCIÓN)
:local debugEnabled false

# Telegram
:local telegramEnabled true
:local telegramToken "TU_BOT_TOKEN"
:local telegramChatId "TU_CHAT_ID"

# WhatsApp (textmebot)
:local whatsappEnabled false
:local whatsappPhone "+549..."
:local whatsappApiKey "TU_API_KEY"

# DDNS WAN1 (cdmon)
:local ddns1Domain "tudominio.com"
:local ddns1User "TU_USUARIO_DDNS"
:local ddns1Hash "MD5_DE_TU_PASSWORD"

# Interfaces (V3 usa nombres con comentario ::)
:local wan1Interface "ether1::Movistar"
:local wan2Interface "ether2::Starlink"

Control de Debug / Log Limpio

💡 Log limpio opcional: El script NO toca el sistema de logging automáticamente (eso ensuciaría el log). Si querés log limpio sin mensajes de fetch, ejecutá manualmente ANTES: 
/system logging set [find topics~"info"] topics=info,!fetch
Y para restaurar: 
/system logging set [find topics~"info"] topics=info

🔐 DDNS con cdmon

Lógica de actualización inteligente

El script consulta primero qué IP está registrada en cdmon, la compara con la IP pública actual, y solo actualiza si difieren. Esto evita llamadas innecesarias a la API.

# 1. Consultar IP registrada actualmente en cdmon
/tool fetch url=("https://dinamico.cdmon.org/onlineService.php?enctype=MD5&n=" . $user . "&p=" . $hash) output=user as-value
# Respuesta contiene "newip=XXX&..." → se parsea

# 2. Obtener IP pública real
/tool fetch url="http://ipinfo.io/ip" src-address=$ipLocal output=user as-value

# 3. Si difieren, actualizar con &cip=
/tool fetch url=("...?enctype=MD5&n=" . $user . "&p=" . $hash . "&cip=" . $nuevaIP) keep-result=no

Obtener el hash MD5

# Linux/Mac
echo -n "tu_password_cdmon" | md5sum

🛰️ CGNAT y WAN sin IP pública

CGNAT (Carrier Grade NAT) es cuando el proveedor te asigna una IP de un rango privado del carrier (ej: Starlink en 100.64.0.0/10) en lugar de una IP pública real. La IP que ve el mundo exterior NO es configurable por vos.

⚠️ Implicancia para DDNS: Si tu WAN2 está en CGNAT, el DDNS es inútil porque no podés apuntar un dominio a esa IP (no es accesible desde afuera, no admite portforwarding ni DMZ). Por eso en V3 el flag wan2DdnsEnabled = false deja WAN2 solo en modo monitoreo.

¿Cuándo habilitar DDNS en WAN2?

  • Starlink plan Business (IP pública dedicada)
  • Otro proveedor que asigna IP pública directa
  • Modem con portforwarding/DMZ hacia el MikroTik

En esos casos: :local wan2DdnsEnabled true y configurar ddns2*.

📥 Instalación

Como script programado

  1. Winbox → System → Scripts → New
  2. Name: ddns-check, Source: pegar contenido del .txt
  3. System → Scheduler → New:
    • Name: ddns-periodic
    • On Event: ddns-check
    • Interval: 5m
    • Start Time: startup

Test manual

/system script run ddns-check
/log print where topics~script

🔧 Troubleshooting

❓ Failover no ocurre al caer WAN1

/ip route print where gateway="1.1.1.1"
# Si está Inactive:
/ip route set [find gateway="1.1.1.1"] target-scope=11

❓ WAN2 figura sin internet pero sí hay conexión (V3)

El ping debe salir por la IP de WAN2. Verificá en terminal:

/ping 8.8.8.8 src-address=IP_DE_TU_WAN2 count=3

La IP se obtiene de la address table del interface WAN2.

❓ Error "expected end of command"

  • /tool fetch con interface= o routing-table= → usar src-address=
  • Parseo con :find ... >= 0 → en RouterOS :find devuelve "", no -1

❓ DDNS falla con IP local en vez de pública

No leer la IP del interface directamente para WAN con NAT (devuelve IP privada). Usar fetch a ipinfo.io con src-address correcto.

❓ Logs llenos de "Download from ipinfo.io FINISHED"

/system logging set [find topics~"info"] topics=info,!fetch
